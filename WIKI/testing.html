🧪 Testing Guide

Retningslinjer for skrivning og kørsel af tests i Flutter Calories projektet.

🎯 Oversigt over Testning

Testning er en kritisk del af udviklingsprocessen. Det sikrer kodens kvalitet, reducerer regressionsfejl og gør refaktorering mere sikker. Vi anvender primært tre typer tests:

  • Unit Tests: Tester individuelle funktioner, metoder eller klasser.
  • Widget Tests: Tester Flutter widgets isoleret (UI-komponenter).
  • Integration Tests: Tester komplette app flows eller større dele af appen.

Den generelle teststrategi er også beskrevet i Arkitektur Dokumentationen.

🏃 Kørsel af Tests

Tests kan køres fra terminalen i rodmappen af projektet.

  • Kør alle Unit og Widget Tests: 
    flutter test

    Dette kommando vil køre alle filer der ender med _test.dart i test/ mappen.

  • Kør en specifik fil: 
    flutter test test/features/onboarding/application/onboarding_notifier_test.dart
  • Kør Integration Tests:

    Integration tests ligger typisk i mappen integration_test/. For at køre dem:

    flutter test integration_test

    Dette kræver en tilsluttet enhed eller emulator. Sørg for at integration_test pakken er tilføjet til dev_dependencies i pubspec.yaml hvis du vil skrive dem.

✍️ Skrivning af Tests

Følg disse retningslinjer når du skriver tests.

Generelle Principper

  • Placering: Testfiler skal placeres i test/ mappen og spejle mappestrukturen i lib/. For eksempel, en test for lib/features/auth/auth_service.dart skal være i test/features/auth/auth_service_test.dart.
  • Navngivning: Testfiler skal ende med _test.dart. Testbeskrivelser (i test() eller group()) skal være klare og beskrive hvad der testes.
  • AAA Pattern: Strukturer dine tests med Arrange, Act, Assert.
    • Arrange: Sæt testdata og mocks op.
    • Act: Udfør den handling eller metode der skal testes.
    • Assert: Verificer at resultatet er som forventet.

Unit Tests

Tester logik i services, notifiers (state management), modeller, og utility funktioner.

Widget Tests

Tester UI komponenter (widgets). Sikrer at UI'en reagerer korrekt på interaktioner og state ændringer.

  • Brug flutter_test pakken og WidgetTester.
  • Test for tilstedeværelse af UI elementer, tekst, og at interaktioner (tryk, input) udløser de forventede ændringer.
  • For widgets der afhænger af Riverpod providers, wrap dem i en ProviderScope og brug overrides til at mocke providers. Se State Management Testing.
  • Eksempel (konceptuelt): 
    // test/widgets/custom_button_test.dart
import 'package:flutter/material.dart';
import 'package:flutter_test/flutter_test.dart';
// import 'package:calories/widgets/custom_button.dart'; // Din widget

void main() {
  testWidgets('CustomButton displays text and handles tap', (WidgetTester tester) async {
    // Arrange
    String buttonText = 'Tryk her';
    bool tapped = false;
    await tester.pumpWidget(
      MaterialApp( // Nødvendig for at give widget'en context (Material theming)
        home: Scaffold(
          body: CustomButton(
            text: buttonText,
            onPressed: () {
              tapped = true;
            },
          ),
        ),
      ),
    );

    // Act
    expect(find.text(buttonText), findsOneWidget);
    await tester.tap(find.byType(CustomButton));
    await tester.pump(); // Rebuild widget efter interaktion

    // Assert
    expect(tapped, isTrue);
  });
}

Golden Tests (Visuel Regression)

Golden tests sammenligner dine widgets med et "gyldent" referencebillede for at fange utilsigtede visuelle ændringer. Pakken golden_toolkit kan bruges til dette.

  • Nyttigt for at sikre UI konsistens over tid.
  • Kræver at man genererer og vedligeholder referencebilleder.
  • Kørsel: flutter test --update-goldens for at opdatere referencebilleder.

Integration Tests

Tester hele features eller brugerflows. Kører på en enhed eller emulator.

  • Brug integration_test pakken. Tilføj den til dev_dependencies: flutter pub add dev:integration_test. Opret testfiler i integration_test/ mappen (f.eks. integration_test/app_test.dart).
  • Simuler brugerinteraktioner (tryk, scroll, input) og verificer UI tilstande og dataflow.

📊 Kode Dækning (Code Coverage)

Kode dækning måler hvor stor en procentdel af din kode der er dækket af tests.

  • Generer Rapport: 
    flutter test --coverage
  • Rapporten genereres i coverage/lcov.info.
  • For at se en HTML rapport, kan du bruge genhtml (en del af lcov værktøjet) eller en IDE extension (f.eks. "Coverage Gutters" i VS Code). 
    # Eksempel på generering af HTML rapport (kræver lcov installeret)
genhtml coverage/lcov.info -o coverage/html
  • Mål: Stræb efter høj kode dækning, især for kritisk forretningslogik (domain og application lag). Der er ikke et fastsat %-mål for projektet endnu, men god dækning er opfordret.

⭐ Bedste Praksis for Testning

  • Skriv tests samtidig med, eller lige efter, du skriver ny kode.
  • Test for både succes-scenarier og fejl-scenarier.
  • Hold tests små og fokuserede på én ting.
  • Gør tests uafhængige af hinanden. En tests resultat må ikke påvirke en anden.
  • Brug beskrivende navne for testgrupper og individuelle tests.
  • Refaktorer tests ligesom du refaktorerer produktionskode for at holde dem rene og vedligeholdelsesvenlige.
  • Kør tests ofte, især før du committer kode.
