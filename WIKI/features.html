✨ Feature Struktur Guide

Retningslinjer for organisering og udvikling af features i Flutter Calories appen.

🎯 Introduktion

Dette dokument beskriver den anbefalede struktur for feature-moduler i projektet. Formålet er at sikre konsistens, vedligeholdbarhed og skalerbarhed i overensstemmelse med Clean Architecture principperne.

Alle features er organiseret som separate moduler under mappen lib/features/.

🏛️ Feature Modul Struktur (Lag)

Hver feature-modul (f.eks. lib/features/user_profile/) bør indeholde følgende undermapper, der repræsenterer de forskellige lag i Clean Architecture:

lib/
└── features/
    └── [feature_name]/
        ├── domain/         # Forretningslogik og -regler (kerne-modeller, interfaces)
        ├── application/    # Applikationsspecifik logik (state management, use case koordination)
        ├── infrastructure/ # Implementering af eksterne afhængigheder (repositories, services)
        └── presentation/   # UI-komponenter (sider, widgets)

🔵 Domain Layer (domain/)

Kernen i din feature. Indeholder ren forretningslogik og er uafhængig af UI og eksterne services.

  • Entities/Models: Data-strukturer der repræsenterer kernekoncepter for featuren (f.eks. UserProfileModel.dart). Disse er typisk simple Dart-klasser eller Freezed-modeller.
  • Use Cases (Abstrakte): Definerer operationer som featuren kan udføre (f.eks. abstract class GetUserProfileUseCase {...}). Ofte unødvendigt for simple features, hvor Application-laget direkte bruger repositories.
  • Repository Interfaces (Abstrakte): Kontrakter for hvordan data hentes og gemmes (f.eks. abstract class UserProfileRepository { Future<UserProfile> getUserProfile(); }).
  • Værdiobjekter (Value Objects): Små, simple objekter hvis lighed ikke er baseret på identitet (f.eks. EmailAddress, Password).
Vigtigt: Domain-laget må ikke have afhængigheder til Flutter SDK (UI) eller andre lag (application, infrastructure, presentation). Det skal være ren Dart.

🟢 Application Layer (application/)

Orkestrerer dataflow og håndterer state for featuren. Forbinder Presentation-laget med Domain- og Infrastructure-lagene.

  • State Management (Notifiers/Providers): Logik for at håndtere og opdatere UI-state (f.eks. Riverpod StateNotifier klasser som UserProfileNotifier.dart).
  • State klasser: Definerer strukturen af den state, som UI'en observerer (f.eks. UserProfileState.dart, ofte Freezed-klasser).
  • Use Case Implementeringer (hvis abstrakte use cases bruges): Konkrete implementeringer af use cases defineret i domain-laget. For simplere features kan Notifiers direkte kalde repository-metoder.
  • Event Handlers/Logik: Håndterer brugerinput og andre events fra UI.

🟠 Infrastructure Layer (infrastructure/)

Implementerer de interfaces (kontrakter) der er defineret i Domain-laget. Håndterer kommunikation med eksterne kilder som databaser, netværks-API'er, og platformsservices.

  • Repository Implementeringer: Konkrete implementeringer af repository interfaces (f.eks. UserProfileRepositoryImpl.dart der bruger SharedPreferences eller en API-service).
  • Data Transfer Objects (DTOs): Modeller der bruges til at serialisere/deserialisere data fra/til eksterne kilder (f.eks. JSON fra en API). Disse mappes typisk til/fra Domain-modeller. Placeres ofte i en dtos/ undermappe.
  • Service Wrappers: Klasser der interagerer direkte med eksterne services (f.eks. FirebaseAuthService.dart, OpenFoodFactsApiService.dart).
  • Platform-specifik kode: Hvis featuren kræver adgang til native funktionalitet.

🟣 Presentation Layer (presentation/)

Alt relateret til brugergrænsefladen (UI).

  • Pages/Screens: Top-level widgets der repræsenterer en hel skærm i appen (f.eks. UserProfilePage.dart).
  • Widgets: Genbrugelige UI-komponenter specifikke for featuren (f.eks. profile_avatar_widget.dart). Disse placeres ofte i en widgets/ undermappe.
  • Navigation: Håndtering af navigation til og fra featurens skærme.
  • UI State Observation: Lytter til state ændringer fra Application-laget (f.eks. via Riverpod ConsumerWidget) og opdaterer UI'en derefter.

🛠️ Sådan Oprettes et Nyt Feature Modul

  1. Opret Hovedmappe:

    Under lib/features/, opret en ny mappe med et beskrivende, snake_case navn for din feature (f.eks. lib/features/meal_planner/).

  2. Opret Lag-Undermapper:

    Indeni din nye feature-mappe, opret undermapperne: domain, application, infrastructure, og presentation.

  3. Definer Domain Lag:
    • Opret dine core entities/modeller i domain/.
    • Definer repository interfaces (abstrakte klasser) for datahåndtering.
  4. Implementer Application Lag:
    • Opret Riverpod Notifiers (eller anden state management logik) i application/.
    • Definer de tilhørende state-klasser (ofte med Freezed).
    • Implementer logik der bruger domain repository interfaces til at hente/manipulere data.
  5. Implementer Infrastructure Lag:
    • Opret konkrete repository implementeringer i infrastructure/. Disse klasser vil implementere interfaces fra domain-laget og interagere med databaser, API'er osv.
    • Opret DTOs hvis nødvendigt for at mappe eksterne dataformater.
  6. Byg Presentation Lag:
    • Opret sider/skærme og specifikke widgets i presentation/.
    • Brug Riverpod til at forbinde UI med state fra application-laget.
  7. Registrer Providers:

    Sørg for at dine Riverpod providers er korrekt registreret og tilgængelige. Dette kan gøres globalt eller mere lokalt afhængigt af providerens scope.

  8. Test Hvert Lag:

    Skriv unit tests for domain og application logik, og widget tests for presentation-laget. Se Testing Guiden.

🌐 Delt Kode

Ikke al kode hører til en specifik feature. Projektet har også følgende mapper for delt kode:

  • lib/core/: Indeholder app-brede utilities, konstanter, tema, navigation, og core infrastructure services (f.eks. generisk storage service) der ikke er specifikke for en enkelt feature.
  • lib/shared/: Indeholder delte widgets eller utility funktioner der kan genbruges på tværs af flere features, men som ikke er fundamentale nok til at ligge i core/.

Overvej nøje om ny kode er feature-specifik eller om den bør placeres i core/ eller shared/.
