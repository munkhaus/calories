📝 Kodestandard og Retningslinjer

For at sikre konsistens og høj kvalitet i koden, følg venligst disse standarder.

📏 Kodeformatering

Al Dart kode skal formateres med den indbyggede Dart formattering.

  • Automatisk Formatering:

    De fleste IDE'er (VS Code, Android Studio) kan sættes op til at formatere koden automatisk ved gemning. Dette anbefales kraftigt.

    I VS Code kan du tilføje følgende til dine `settings.json`:

    "[dart]": {
    "editor.formatOnSave": true,
    "editor.formatOnType": true,
    "editor.rulers": [80], // Anbefalet linjelængde
    "editor.selectionHighlight": false,
    "editor.suggest.snippetsPreventQuickSuggestions": false,
    "editor.suggestSelection": "first",
    "editor.tabCompletion": "onlySnippets",
    "editor.wordBasedSuggestions": false
}
  • Manuel Formatering:

    Kør følgende kommando i terminalen fra projektets rodmappe:

    dart format .
  • Linjelængde: Stræb efter en maksimal linjelængde på 80 tegn for at forbedre læsbarheden. IDE'er kan konfigureres til at vise en guide-linje.

🏷️ Navngivningskonventioner

Følg de officielle Dart Naming Conventions.

  • Klasser, Enums, Typedefs, Extensions: UpperCamelCase (f.eks. UserProfile, AuthStatus).
  • Metoder, Funktioner, Variabler, Parametre: lowerCamelCase (f.eks. getUserProfile, userName).
  • Konstanter (compile-time constants): lowerCamelCase eller SCREAMING_SNAKE_CASE hvis de er globale/static og repræsenterer en dyb, uforanderlig værdi (f.eks. kDefaultTimeout, MAX_RETRIES). Foretræk lowerCamelCase for de fleste konstanter.
  • Filer og Mapper: snake_case (f.eks. user_profile_page.dart, auth_service.dart). Undtagelser er filer der kun indeholder én klasse, hvor filnavnet kan være klassens navn i snake_case (f.eks. hvis klassen er `UserProfile`, kan filen hedde `user_profile.dart`).
  • Private medlemmer: Start navnet med et underscore (_) (f.eks. _privateMethod, _internalCounter).
  • Booleans: Præfiks ofte med `is`, `has`, `should`, `can` (f.eks. `isLoading`, `hasConnection`).

🔍 Linting

Projektet bruger package:flutter_lints/flutter.yaml som grundlag for statisk analyse. Dette sæt af lints hjælper med at identificere potentielle fejl, stilproblemer og anti-patterns.

  • Konfiguration: Lint reglerne er defineret i analysis_options.yaml filen.
  • Kør Linter:

    Du kan analysere koden ved at køre følgende kommando:

    flutter analyze

    Eller blot dart analyze.

  • IDE Integration: VS Code og Android Studio viser linting advarsler og fejl direkte i editoren. Sørg for at løse disse, før du committer kode.
  • Undgå // ignore: lint_rule: Brug kun ignore kommentarer når det er absolut nødvendigt og efter diskussion med teamet. Inkluder altid en forklaring på hvorfor reglen ignoreres.

💡 Generelle Retningslinjer

  • Følg Clean Architecture: Respekter lagdelingen (Domain, Application, Infrastructure, Presentation) som beskrevet i Arkitektur Dokumentationen. Placer filer og logik i de korrekte lag.
  • Immutability: Foretræk uforanderlige (immutable) datastrukturer, især for state klasser (f.eks. ved brug af Freezed).
  • Single Responsibility Principle (SRP): Klasser og metoder bør have ét specifikt ansvar.
  • DRY (Don't Repeat Yourself): Undgå kodegentagelse ved at genbruge logik og widgets hvor det er muligt.
  • Kommentarer:
    • Skriv kommentarer for kompleks logik, eller hvor koden ikke er selvforklarende.
    • Brug /// for dokumentationskommentarer til offentlige API'er (klasser, metoder).
    • Undgå overflødige kommentarer der blot gentager hvad koden gør.
  • Widgets:
    • Opdel store widgets i mindre, genbrugelige komponenter.
    • Brug const constructors for widgets hvor det er muligt for at forbedre performance.
    • Placer build-metoder så tæt på deres brug som muligt for at undgå unødvendige rebuilds.
  • Asynkron Kode:
    • Brug async/await for læsbar asynkron kode.
    • Håndter fejl i Futures og Streams korrekt (f.eks. med try-catch eller .catchError()).
    • Overvej brugen af Result type for at håndtere succes/fejl tilstande eksplicit.
  • Error Handling: Implementer robust fejlhåndtering. Vis brugervenlige fejlmeddelelser i stedet for at lade appen crashe.
  • Testning: Skriv tests for din kode som beskrevet i Testing Guiden.
  • Dependencies: Vær forsigtig med at tilføje nye dependencies. Overvej om funktionaliteten kan opnås med eksisterende pakker eller Dart/Flutter SDK'en.

📦 Commit Beskeder

Følg Conventional Commits standarden for commit beskeder. Dette hjælper med at gøre historikken mere læsbar og muliggør automatisk generering af changelogs.

Format: type(scope): subject

  • Types: feat (ny feature), fix (bug fix), docs (dokumentation), style (formatering), refactor, test, chore (build-relateret, etc.).
  • Scope (valgfri): Modul eller del af appen der er påvirket (f.eks. onboarding, food_logging).
  • Subject: Kortfattet beskrivelse af ændringen, start med lille bogstav, ingen punktum til sidst.

Eksempel:

feat(onboarding): add physical info step to user setup
fix(food_logging): correct calorie calculation for custom portions
docs(wiki): update setup instructions
style: format all dart files
refactor(auth): simplify token refresh logic

For større ændringer, inkluder en krop i commit beskeden med mere detaljeret information.
